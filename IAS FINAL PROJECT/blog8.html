What is SQL Injection?

SQL Injection (SQLi) is a critical security weakness (CWE-89). It occurs when applications build SQL commands using user input without properly sanitizing special characters. This allows attackers to inject their own SQL code and manipulate the intended query.

The Core Problem

Applications often construct SQL queries by concatenating strings, including user input. Special SQL characters (quotes, semicolons, dashes) can break query logic and allow new commands.

Example: Login Bypass

Intended Query: SELECT * FROM users WHERE username = 'USER_INPUT' AND password = 'PASSWORD_INPUT';

Malicious Input: ' OR '1'='1

Resulting Query: SELECT * FROM users WHERE username = '' OR '1'='1' AND password = 'PASSWORD_INPUT';

Effect: The condition '1'='1' is always true, granting unauthorized access.

Why SQLi is Dangerous

  • Data Breach: Steal sensitive information like credentials or personal data.
  • Data Loss/Corruption: Modify or delete database content.
  • Denial of Service (DoS): Overload the database, making it unavailable.
  • Full System Compromise: Take control of the database or web server.

Mitigation Strategies

The goal is to prevent user input from being treated as executable SQL code:

  • Prepared Statements: Compile the query first, then insert user input as data. This is the most effective defense.
  • Input Validation: Check input patterns, length, and allowed characters to reject malicious input early.
  • Stored Procedures: Use pre-compiled SQL code to limit dynamic SQL construction.
  • Escaping User Input: Add escape characters before special SQL characters. Less secure and error-prone than prepared statements.
  • Principle of Least Privilege: Limit database account privileges to reduce potential damage.
